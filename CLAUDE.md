## Overview

This is a GitHub Action that automatically creates and manages Veracode teams
based on repository configurations. It integrates with the Veracode Identity API
to ensure teams exist and are properly configured with validated members.

## Key Architecture

### Repository Structure

```
src/
├── main.ts              # Action entry point - orchestrates workflow
├── index.ts             # Exports run() for GitHub Actions
├── types.ts             # Type definitions and interfaces
├── errors.ts            # Error handling and categorization
├── utils.ts             # Utility functions
├── config/
│   ├── resolver.ts      # Maps repositories to team configurations
│   └── validator.ts     # Validates YAML config using Zod schemas
├── services/
│   ├── team-service.ts  # Veracode team operations (lookup, create, update)
│   ├── user-validator.ts # Validates team members against Veracode platform
│   └── github-service.ts # Fetches GitHub collaborators
├── veracode/
│   ├── client.ts        # Axios-based HTTP client for Veracode Identity API
│   └── auth.ts          # HMAC-SHA256 authentication
└── utils/
    ├── retry.ts         # Exponential backoff retry logic
    └── rate-limiter.ts  # Request rate limiting
```

### Action Workflow (main.ts)

The action follows this sequence:

1. **Input Validation**: Retrieves and validates GitHub Actions inputs
2. **Client Initialization**: Sets up Veracode and GitHub clients
3. **Configuration Loading**: Fetches and validates team mapping YAML from
   config repository
4. **Team Resolution**: Resolves repository name to team configuration (supports
   wildcards)
5. **GitHub Sync** (optional): Fetches GitHub collaborators if
   `sync_github_collaborators: true`
6. **Member Validation**: Validates all users exist and are active in Veracode
7. **Team Lookup**: Checks if team already exists
8. **Team Operation**: Creates new team or updates existing team with valid
   members
9. **Output & Summary**: Sets GitHub Actions outputs and writes step summary

### Configuration System

- **Resolver** (`config/resolver.ts`): Matches repository names against
  configured patterns (exact, wildcard `*`) and returns the team configuration
- **Validator** (`config/validator.ts`): Uses Zod to validate the YAML mapping
  file structure
- **Fallback**: Supports optional auto-creation of teams for unmapped
  repositories using templates

### Data Flow

1. Team mapping YAML → Validator (Zod schema) → Resolver
2. Repository name → Resolver → TeamConfiguration
3. TeamConfiguration → UserValidator → Valid members only
4. Valid members → TeamService → Veracode API → Team ID

### Error Handling

- **ErrorCategory enum**: Categorizes errors as CONFIGURATION, AUTHENTICATION,
  VALIDATION, or API
- **RetryLogic**: Exponential backoff with configurable retries for transient
  failures
- **Sanitization**: Log messages sanitize sensitive IDs/names for security

## Important Conventions

### TypeScript & Bundling

- **Sources in `src/`**: All TypeScript source files live here
- **Built output in `dist/`**: JavaScript bundle generated by Rollup (do NOT
  edit manually)
- **Always bundle after changes**: Run `npm run bundle` after editing `src/` to
  update `dist/`
- **GitHub workflow check**: The CI validates that `dist/` is up-to-date with
  `src/`

### Testing

- **Unit tests in `__tests__/`**: Jest-based tests
- **Fixtures in `__fixtures__/`**: Test data and mock responses
- **Minimum 80% coverage required**: Enforced in CI
- **Test both happy path and edge cases**: Configuration validation, missing
  users, API failures, etc.

### Logging

- **Use `@actions/core`** for all logging (`core.info()`, `core.warning()`,
  `core.error()`)
- **Never use `console`** - GitHub Actions requires @actions/core for proper log
  formatting
- **Sanitize logs**: Sensitive information (IDs, emails, secrets) must be
  sanitized with `sanitizeForLog()`

### API Integration

- **HMAC-SHA256 auth**: Implemented in `veracode/auth.ts` - generates
  Authorization headers
- **Paginated responses**: Identity API uses pagination; code handles up to 100
  pages
- **Rate limiting**: Rate limiter in `utils/rate-limiter.ts` prevents throttling
- **Error categorization**: All API errors are categorized for better debugging

### Type Safety

- **Zod schemas**: Single source of truth for config validation (defined in
  `config/validator.ts`)
- **Type exports from validator**: All TeamConfiguration types are exported from
  config/validator.ts
- **No `any` types**: Use proper interfaces and types

## Common Tasks

### Adding a New Input

1. Add to `action.yml` under `inputs`
2. Add to `ActionInputs` interface in `types.ts`
3. Update `getInputs()` in `main.ts` to retrieve it with `core.getInput()`
4. Use the new input in the appropriate service

### Adding a New Output

1. Add to `action.yml` under `outputs`
2. Add to `ActionOutputs` interface in `types.ts`
3. Update `setOutputs()` in `main.ts` to call `core.setOutput()`

### Modifying Configuration Schema

1. Update the Zod schema in `config/validator.ts`
2. Export the new type from validator.ts
3. Update type imports in `types.ts` if needed
4. Update team mapping examples in README.md

### Adding Veracode API Endpoints

1. Add method to `VeracodeClient` in `veracode/client.ts`
2. Include error categorization and retry logic
3. Document parameters and return types
4. Add corresponding service method if needed (e.g., in
   `services/team-service.ts`)
